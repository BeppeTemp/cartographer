# MCP transport and authentication

## Transport

| Transport | Profile | Details |
|---|---|---|
| **stdio** | Local Core | Single agent. No token. Signing identity from local configuration (env/CLI or git identity). Newline-delimited JSON-RPC 2.0. |
| **Streamable HTTP** | Server | Canonical, containerizable transport. A single endpoint (e.g. `/mcp`): `POST` (JSON-RPC; `application/json` or `text/event-stream` response) and `GET` (optional SSE stream). |

Do not expose the legacy two-endpoint HTTP+SSE transport (deprecated as of MCP revision 2025-03-26) except as a documented shim.

## Authentication (Server profile)

**OAuth 2.1 resource-server**:
- Bearer in the `Authorization` header on **every** request (never in the query string).
- Audience binding on the canonical URI.
- `401` with `WWW-Authenticate` → `/.well-known/oauth-protected-resource` (Protected Resource Metadata, RFC 9728).
- Token issuance **delegated to an external AS**.
- Auto-registration via **Client ID Metadata Documents** (preferred); DCR allowed as a fallback.

Locally (stdio) or on a trusted network: bearer token generated by the server's CLI.

Auth can be explicitly enabled/disabled via `CARTOGRAPHER_AUTH` (see [`deployment.md`](deployment.md) §Environment variables).

## Per-KB authorization (r/rw scope, HTTP enforcement)

The token carries a **per-KB scope** listing the accessible KBs and the right (`r`/`rw`).

Format: `kb:<name>:<r|rw>`, space or `;` separated (`auth.ParseScopes`). Example: `kb:homelab:rw kb:archivio-storico:r`.

`<name>` is the KB name: `kbs[].name` explicit if set (D53), otherwise the **basename**
of the directory/repo (last path segment, `.git` suffix removed) — the same derivation used
for `?kb=<name>` routing and for the clone directory. A token's scope must match
this name exactly.

**Token configuration with scope** (YAML `auth.tokens` or `CARTOGRAPHER_TOKENS`/`--tokens`):
- YAML: each entry is `{token: ..., scopes: ["kb:homelab:rw", "kb:archivio-storico:r"]}` (`config.TokenSpec`); a bare scalar (string) remains supported for backward compatibility and means full access.
- `CARTOGRAPHER_TOKENS`/`--tokens`: `token|kb:homelab:rw;kb:archivio-storico:r`, multiple entries separated by comma/space.
- **No scope (`scopes` absent/empty) = full access (admin)** to all mounted KBs: default behavior, backward-compatible with plain pre-M2 tokens.

**Enforcement** (`internal/auth`, `internal/mcpserver/httpserver.go`):
1. `auth.TokenStore` (created via `auth.NewScopedTokenStore`) maps the token hash → `[]auth.KBScope`. `TokenStore.Middleware` validates the bearer and injects the scopes into the context (`auth.ContextWithScopes`/`ScopesFromContext`).
2. The HTTP guard (`mcpAccessGuard` in `httpserver.go`, downstream of the Middleware, for each KB mounted at `/mcp?kb=<name>`) reads the scopes from the context: **empty/nil scopes → always pass** (full access). Otherwise it peeks at the JSON-RPC body (`io.LimitReader` 2MB) to decide whether the request needs `r` or `rw`, then **always restores `r.Body`** (`io.NopCloser` over the read bytes) so the downstream handler re-reads it intact.
3. r/rw decision: `method != "tools/call"` (initialize, tools/list, ping, ...) → requires `r`; `tools/call` → requires `rw` if the tool is not marked `ReadOnly` (see `ToolRequiresWrite` in `internal/mcpserver/readonly.go`), otherwise `r`. **Fail-closed**: unparsable body or unknown tool → requires `rw`.
4. Final check with `auth.HasAccess(scopes, kbName, needWrite)`; if denied → `403 forbidden` (`auth.Forbidden`).

**Read-only tool classification**: exhaustive list with `Tool.ReadOnly=true` in `internal/mcpserver` → see [`control-plane.md`](control-plane.md) §API MCP. All other tools (governance/write/skill_install/sync_apply/git_conflict_resolve/...) require `rw`.

In the **Local Core** (stdio, single-user) no ACL is needed: no token, no scope.

> Do not mount KBs of mixed sensitivity on the same instance without an active scope on the tokens.

## Identity → signature

The token/OAuth (or the local identity for stdio) signs commits and log entries. The audit log is append-only and tamper-evident (hash-chain on JSONL). See [`control-plane.md`](control-plane.md) §audit.

## Compliance-grade audit

The audit log (`internal/audit`) is append-only JSONL with SHA-256 hash-chain. Each entry includes `prev_hash` and `hash`; `Verify()` checks the entire chain.

**Ed25519 signature (opt-in)**: if `CARTOGRAPHER_AUDIT_KEY` is set (32-byte hex seed = 64 characters), every entry is signed with Ed25519. The `sig` field (hex, omitempty) holds the signature of `hash`. Old logs without `sig` remain valid (backward-compatible).

```
CARTOGRAPHER_AUDIT_KEY=<64-char hex seed>   # generate with: openssl rand -hex 32
```

Relevant API: `audit.OpenWithKey(path, kp)` to enable signing; `Log.Verify()` verifies chain + signature; `Log.VerifyFull()` distinguishes signed, unsigned and invalidly-signed entries.

> The key must be kept separate from the log: with it the integrity is verifiable; without it, the log remains readable but not cryptographically verifiable.

## Stateless-friendly

Do not depend on `Mcp-Session-Id` (being removed in MCP RC 2026-07-28). Locks and persisted state live outside the versioned content; `if_match` works without sessions.
