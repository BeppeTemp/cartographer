# Interoperability: OKF, providers and the configurator

## OKF conformance

Every KB is a **conformant OKF v0.1 bundle** *and* a git repository in its own right. Any external OKF tool can consume it without modification. Everything is **git + files**: history, diff, backup, sharing, zero lock-in. The skills (`SKILL.md` agentskills.io) and secrets (SOPS) are likewise open, portable standards.

**Obsidian** is the human viewer (graph view, Dataview), conceptually read-only.

## Hard rules vs soft guidance

- **Hard rules тЖТ server**: validation, gate, raw immutability. The KB is always mediated by the server via MCP tools: no agent edits it directly on the filesystem.
- **Soft guidance тЖТ `instructions` provisioning** (D56/D61, no longer an `AGENTS.md` in the KB тАФ removed by D62): a generated block injected into each provider's global instructions file (`.claude/CLAUDE.md`, `.config/opencode/AGENTS.md`, `.codex/AGENTS.md`, Kiro steering), plus the optional curated `instructions.md` in the KB root (D61) and the auto-generated agent section тАФ see `sync.md` ┬зInstructions.

## The MCP + skill + hook triad

| Level | What it does | Token cost |
|---|---|---|
| **MCP** | Data and capabilities (bounded reads, search, validated writes) | Proportional to the content read |
| **Skill** | Competence/procedure (how to do ingest/query/lint), loaded on-demand | ~100 tokens catalog; body on-demand |
| **Hook** | Deterministic automation (post-ingest тЖТ scoped lint; pre-commit тЖТ commit_gate) | 0 тАФ uses no reasoning |

## Multi-provider configurator

A **dedicated tool** (the `cartographer` client subcommands тАФ `agents`/`connect`/`status`/`sync`,
see `docs/configurator.md`) configures skills, MCP, hooks (and references to services/secrets) on
each provider leveraging its capabilities to the **fullest** тАФ no least-common-denominator. The
client always talks to the server over HTTP (no stdio transport on the client side, `decisions.md` ┬зD37).

**Single source-of-truth тЖТ per-provider adapter**: the source is the **provisioning manifest** generated by the server (compiled bundle + artifacts of the mounted KBs тАФ skills, agents, hooks, instructions); a per-provider adapter (`internal/configurator` for the MCP config, `internal/provisioning.Apply` for materialization) translates each artifact into the native constructs, discarding what cannot be reliably mapped instead of guessing it. The full kind├Чprovider matrix is in `sync.md` ┬зKind ├Ч provider matrix.

**TUI**: interactive dashboard (`cartographer` with no arguments): autodetects installed providers, per-provider status with per-kind counts, connect/resync/disconnect тАФ see `configurator.md` ┬зTUI mode.

**Idempotent and non-destructive sync**: deep-merge for JSON; marker-delimited blocks for TOML and instruction files; never overwrite the user's contracts or secrets; dry-run.

**Automatic realignment** when skills/agents/hooks change (bundle or KB): manifest+revision server-side, lockfile client-side, drift detection and layered triggers (`SessionStart` hook, MCP tool, push). See тЖТ [`sync.md`](sync.md).

**Secrets**: only variable names translated into the native syntax (`${VAR}` Claude/Kiro, `bearer_token_env_var` Codex, `{env:VAR}` OpenCode) + native OAuth storage. `.env.example` generated.

## Known configurator risks

- Divergent MCP transport (OpenCode SSE-first; Kiro no custom header) тЖТ `mcp-remote`/OAuth.
- Kiro in 2.xтЖТV3 transition (the adapter detects the version).
- Codex is more rigid (hooks only `command`).
- Diverging skill frontmatter caps across providers.
- Idempotence on mixed user/tool files (require robust markers or import).
- Non-uniform sandboxing; heterogeneous enterprise governance (four mechanisms).

> Every third-party tool/connector (SOPS, age, `mcp-remote`, synchronizers) must be evaluated for **license** and **supply-chain security** before adoption. Skills are **executable code** and require vetting and workspace trust.
